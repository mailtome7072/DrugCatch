# writing-plans 스킬

스프린트 계획 문서를 작성하는 방법론과 형식을 정의합니다.

---

## 문서 저장 경로

`docs/sprint/sprint{N}.md`

---

## 필수 포함 섹션

### 1. 스프린트 개요

```markdown
## 스프린트 개요

- **스프린트 번호**: Sprint {N}
- **목표**: {한 문장으로 요약한 스프린트 목표}
- **기간**: {시작일} ~ {종료일}
- **담당**: {담당자 또는 팀}
```

### 2. 구현 범위

포함/제외 항목을 명확히 구분합니다.

```markdown
## 구현 범위

### 포함
- {구현할 기능 1}
- {구현할 기능 2}

### 제외 (다음 스프린트로 이월)
- {제외 항목}
```

### 3. 작업 분해 (Task Breakdown)

각 태스크는 독립적으로 완료 가능한 단위로 분해합니다.

```markdown
## 작업 분해

| # | 태스크 | 우선순위 | 예상 시간 | 상태 |
|---|--------|----------|-----------|------|
| 1 | {태스크명} | 높음/중간/낮음 | {시간}h | ⬜ |
```

우선순위 기준:
- **높음**: 스프린트 목표 달성에 필수
- **중간**: 품질 향상 또는 사용자 경험 개선
- **낮음**: 있으면 좋지만 없어도 무방

### 4. 기술적 접근 방법

각 태스크의 구체적인 구현 전략을 기술합니다.

```markdown
## 기술적 접근 방법

### {태스크명}
- 접근법: {구현 전략}
- 관련 파일: `{파일 경로}`
- 주의사항: {잠재적 문제 또는 고려사항}
```

### 5. 의존성 및 리스크

```markdown
## 의존성 및 리스크

| 항목 | 유형 | 대응 방안 |
|------|------|----------|
| {항목} | 의존성/리스크 | {대응} |
```

### 6. 완료 기준 (Definition of Done)

스프린트 성공 기준을 측정 가능하게 정의합니다.

```markdown
## 완료 기준

- [ ] {측정 가능한 완료 조건 1}
- [ ] {측정 가능한 완료 조건 2}
- [ ] `docker compose up --build` 성공
- [ ] 로컬 브라우저에서 주요 기능 동작 확인
```

### 7. 예상 산출물

```markdown
## 예상 산출물

- {결과물 1} (예: 특정 화면 구현 완료)
- {결과물 2} (예: API 엔드포인트 동작)
```

---

## 작성 원칙

1. **구체성**: "UI 구현" 대신 "동의 체크박스 및 확인 버튼 구현 (React 컴포넌트)"처럼 구체적으로
2. **실행 가능성**: 각 태스크는 하루 이내에 완료 가능한 단위로 분해
3. **측정 가능성**: 완료 기준은 "잘 됨"이 아닌 "버튼 클릭 시 다음 화면 전환됨"처럼 검증 가능하게
4. **ROADMAP 정합성**: ROADMAP.md의 해당 스프린트 설명과 일치해야 함
